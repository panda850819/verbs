# 中文寫作報告腔替換表

> 載入觸發: 中文 prose 出現「主要敘事/系統梳理/核心結論是/公開口徑/共同模式已經很穩定/可以從三個角度看」等彙報式表達, 或被指定為「技術部落格」「對外文章」「給同行看的」.
> 來源: Waza skills/write/references/write-zh.md (報告腔專項 + 默認禁用)

報告腔讓技術文像「彙報材料」而非「同行寫給同行」. 替換要直接, 不要保留報告骨架.

## 直接替換表 (高頻命中)

| 少用 | 推薦 |
|------|------|
| 主要敘事 | 大家通常用...來解釋... |
| 系統梳理 | 從 A 一路講到 B |
| 核心結論是 | 最後會看到 |
| 邏輯分工 / 流程節點 | 為了看分工 / 更細的版本 |
| 公開口徑裡 | 按公開數據 |
| 共同模式已經很穩定 | 能看到幾個共同點 |
| 可以從三個角度看 | 可以先看三件事 |
| 不需要讀成 X | 不只是 X |
| 值得注意的是 | 直接說結論 |
| 綜上所述 | 直接收尾 |
| 隨著...的發展 | (刪掉, 直接進主題) |
| 本文旨在 | (刪掉, 第一句直接寫做了什麼) |
| 進行了深入的探討 | 討論了 |
| 進行了全面的分析 | 分析了 |

## 句式級報告腔 (要重寫不只是換詞)

### 1. 「我們可以從 N 個角度看」

> NO: 「我們可以從三個角度看這個問題: 安全、效能、可維護性」
> OK: 「先看三件事: 安全、效能、可維護性」

直接點名事情, 不用「角度」這個元層級.

### 2. 「核心結論是 X」

> NO: 「核心結論是, X 在 Y 場景下不適用」
> OK: 「最後會看到, X 在 Y 場景下不適用」 / 「結論先講: X 在 Y 場景下不適用」

「核心結論」是會議紀錄用詞, 不是文章用詞.

### 3. 「值得注意的是 X」

> NO: 「值得注意的是, 這個改動會影響舊版本的相容性」
> OK: 「這個改動會影響舊版本的相容性」 / 「順帶一提, 這個改動會影響舊版本的相容性」

值得注意的事情, 直接說即可. 自己標「值得注意」是冗餘.

### 4. 「我們進行了深入的探討」

> NO: 「我們對 X 進行了深入的探討」
> OK: 「我們討論了 X」 / 「拆 X 的時候發現...」

「進行 + 動作名詞」是英文 conduct/perform 直譯, 中文不需要這層.

### 5. 「綜上所述, X 非常重要」

> NO: 「綜上所述, 配置管理對長期維護非常重要」
> OK: 直接刪. 如果文章前面已經講清楚, 結尾不需要重複; 如果沒講清楚, 加一句「綜上所述」也救不了.

## 句子評價句也是報告腔

直接寫對象本身, 不要寫對句子的評價.

> NO: 「這句話解釋不了 X」
> OK: 「X 解釋不了 Y」

> NO: 「這個比喻可能不太準確」
> OK: 「比喻換成 Y 更準」

## 檢測信號 (給 LLM 自查用)

掃 prose 時, 出現以下任一就 flag:

- 任何「主要/核心/關鍵」+ 名詞當開頭 (核心結論/主要敘事/關鍵在於)
- 任何「進行 + 動作名詞」結構 (進行討論/進行分析/進行改造)
- 任何「綜上/總而言之/由此可見」當段首
- 任何「值得 + 形容詞」自我評價 (值得注意/值得思考)
- 任何「我們 / 大家 / 讀者」當主語的元評論句

命中 → 用上表替換, 不要原樣保留.

## 一個原則

> 報告腔的本質是「先聲明結構, 再給內容」. 自然寫作是「直接給內容, 結構讓讀者從內容裡看出來」.

報告腔: 「我們可以從三個角度看. 第一, 安全. ...」
自然: 「先看安全. ...」

差別是: 報告腔花一句話宣告框架, 自然寫作把框架隱在標題或第一句的內容裡, 不另外宣告.
