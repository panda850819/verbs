# 中文寫作翻譯腔 4 套路

> 載入觸發: 中文 prose 出現「接住/擊穿/打穿/扛住」等物理動詞當思考過程, 或形容詞 + 冒號引出內容, 或「X 的 Y 比 Z 更 W」骨架, 或 context/state/cache 等已有中譯的英文詞直接混入.
> 來源: Waza skills/write/references/write-zh.md (#16-19)

中文技術寫作骨架是英文時, 句子結構會帶有翻譯腔. 4 個高頻套路:

## 套路 1: 物理動作動詞當思考過程

把抽象認知過程想成物理動作, 骨架是英文 (catch / pierce / sharp / break / blow up).

常見詞: 接住, 擊穿, 鋒利, 不崩, 不爆, 打穿, 扛住, 收緊, 落地, 推開, 撐不住, 砸進去, 硬剛.

> NO: 「你這幾條我都接住了」「這個論證被擊穿了」「更鋒利的重構:」「上下文不崩、成本不爆」
> OK: 「你這幾條我都收到了」「這個假設不成立」「換一種更準的講法:」「上下文不會亂、成本壓得住」

**自檢方式**: 寫完一段, 把所有動詞圈出來, 凡是中文日常不會這樣用的 (你不會跟朋友說「我接住你的話」), 挨個換掉.

## 套路 2: 形容詞預判 + 冒號引出內容

「更乾淨:」「邏輯很清晰:」「問題很直接:」「結論很明確:」, 形容詞先下判斷, 冒號才引出事實.

問題:
- 抢走读者自己评估的机会
- 形容詞幾乎總是多餘的
- 後面的事實本身就能讓讀者得出那個感受

> NO: 「驗證得比我預期的更乾淨: 上下文不崩、成本不爆、狀態可恢復」
> OK: 「數據把你的感覺證出來了: 上下文不亂、成本壓得住、狀態能恢復」
> OK (更好): 直接刪形容詞那段, 只留事實 — 「上下文不亂, 成本壓得住, 狀態能恢復」

**改法**: 把形容詞那一節刪掉, 只留後半句事實. 如果非常想保留那個形容詞, 多半說明後面內容沒講清楚, 要回頭補內容, 不是補形容詞.

## 套路 3: 抽象名詞做主語 + 形容詞當結論

「X 的 Y 比 Z 更 W」骨架: 主語是抽象名詞 (工程上的現實, 這份數據, 生態的成熟度), 結論是形容詞 (難看, 直接, 漫長), 中間系動詞連接.

讀完不知道到底哪兒難看, 難看在哪個環節.

> NO: 「工程上的現實比這些數字難看」
> OK: 「這些數字只反映了採用面; 真往下看各家怎麼接, 早就對不齊了」

> NO: 「這份數據比實際情況樂觀」
> OK: 「這份數據只看了註冊量, 真實活躍跑去 Discord, 看不到」

**改法**: 讓人、動作、或具體對象做主語, 讓事實自己說話. 凡是碰到「X 的 Y 比 Z 更 W」骨架, 都應該重寫.

## 套路 4: 有穩定中譯的英文詞直接混入

context, state, cache, claim 這類詞有現成中文對譯: 上下文, 狀態, 緩存, 斷言. 留著英文原詞讓讀者每次切換都消耗注意力.

> NO: 「context 不崩、成本不爆、state 可恢復、cache 命中率高」
> OK: 「上下文不會亂、成本壓得住、狀態能恢復、緩存命中得上」

**判斷標準**: 中文技術圈已有通用譯法, 就換中文; 術語在中文圈還未收斂 (如 prompt, embedding, tokenizer, attention head), 保留英文合理.

**例外**: SFT, RL, MoE, GRPO, PPO 這類已成行業通用縮寫, 全程保留縮寫形式.

**常見可替換詞**:
- context → 上下文
- state → 狀態
- cache → 緩存
- claim → 斷言
- runtime → 運行時
- contract → 契約
- pipeline → 流水線
- retry → 重試
- fallback → 後備 / 退路

## 自檢清單

每篇 zh 技術 prose 過 edit 模式時, 對 prose 主體掃一遍:

1. 動詞掃: 有沒有「接住/擊穿/打穿/扛住/不崩/不爆」?
2. 冒號掃: 有沒有「形容詞 +:」引出事實的句式?
3. 主語掃: 有沒有「抽象名詞 + 比 + 形容詞」骨架?
4. 英文詞掃: 已有中譯的英文詞有沒有夾在中文裡?

任何一條命中 → 改, 並 flag 給作者看是哪個套路.
