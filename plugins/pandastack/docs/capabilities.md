# Pandastack Capability Map

Three files define the runtime capability map. Together they let pandastack
skills and orchestration code read a stable contract instead of probing
Claude/Codex/Hermes ad hoc at call time.

## File inventory

| File | Scope | Committed? |
|---|---|---|
| `pandastack.toml` | Desired/default config | Yes — repo |
| `.pandastack/local/capabilities.json` | Detected local state | No — gitignored |
| `~/.pandastack/runtimes.json` | Global runtime inventory | No — user-local |

### `pandastack.toml` (committed)

Desired config and role declarations. This is the single source of defaults.
It lists which runtimes belong to each role and where the generated state files
should be written. Do not put host-specific paths or auth state here.

### `.pandastack/local/capabilities.json` (local, gitignored)

Generated by `scripts/pandastack doctor --write-capabilities`. Contains
detected role state, skill counts, auth status (boolean/enum only — no token
values), limits, and next actions for this machine.

Fields:
- `schema_version` — integer, currently 1
- `generated_at` — ISO-8601 UTC timestamp
- `repo` — absolute path to this repo
- `source` — `"scripts/pandastack doctor"`
- `roles.host` — detected host runtime entries (claude, codex, hermes)
- `roles.worker` — worker role state
- `roles.operator` — operator role state
- `runtimes` — `{cmd: bool}` presence map for all tracked commands
- `skills.counts` — `{core, ext, total}` from manifest.toml
- `auth` — auth status per AI runtime (`unknown|present|missing|unsupported`)
- `limits` — network and write-scope detected bounds
- `next_actions` — actionable items based on detected state

Secret hygiene: auth states are boolean/enum only. No tokens, no env var
values, no keychain entries, no raw CLI output that might leak credentials.

### `~/.pandastack/runtimes.json` (global, user-local)

Generated alongside the local file. Contains the runtime inventory for the
current user across all repos.

Fields:
- `schema_version` — integer, currently 1
- `generated_at` — ISO-8601 UTC timestamp
- `runtimes` — per-AI-runtime entries with `present`, `path`, and `auth`

## Generating the map

```bash
# Write both files
scripts/pandastack doctor --write-capabilities

# Also print capabilities JSON to stdout
scripts/pandastack doctor --write-capabilities --json

# Read current map (from files if present, live detection fallback)
scripts/pandastack doctor --capabilities-json
```

## Reading the map (stable API)

```bash
# Emit merged capability map as JSON
scripts/pandastack doctor --capabilities-json | jq .
```

Priority order for `--capabilities-json`:
1. Committed defaults from `pandastack.toml`
2. Local `.pandastack/local/capabilities.json` if it exists
3. Global `~/.pandastack/runtimes.json` override on top
4. Live PRO-15 detection as fallback when no generated files exist

If local/global files are missing, the output includes a `warnings` array with
actionable messages. The command never crashes on missing files.

## Runtime support boundaries

| Runtime | Host role | Worker backend | Auth probe |
|---|---|---|---|
| Claude Code | Supported | — | `unknown` (no probe) |
| Codex CLI | Supported | Supported (primary) | `unknown` (no probe) |
| Hermes | Supported (via pdctx) | — | `unsupported` if `hermes` not on PATH |

## PRO-18 handoff

PRO-17 establishes this map and provides the stable `--capabilities-json` read
path. PRO-18 will migrate skills and orchestration code to consume the map
instead of probing runtimes ad hoc.
